# Requirements Document

## Introduction

This document specifies requirements for the STMS Simulation Refactor — a major extension of the existing Smart Traffic Management System (STMS) that transforms it from a traffic management tool into a live city traffic simulation platform. The fictional city of Meridian City serves as the setting. The refactor introduces a real interactive map (MapLibre GL + OpenStreetMap as primary, Mapbox GL JS as optional fallback), a server-side algorithmic simulation engine, a scenario/event injection system, a visible AI insight layer, an emergency vehicle priority mode, and a polished command-centre UI. All changes are additive or refactors on a new branch; the existing 113 tests must continue to pass.

---

## Glossary

- **Map_Provider**: The abstraction layer that wraps either MapLibre GL JS or Mapbox GL JS and exposes a unified interface to all map components.
- **MapLibre**: MapLibre GL JS — the default, free/open-source map renderer using OpenStreetMap tiles.
- **Mapbox**: Mapbox GL JS — the optional premium map renderer, activated only when `NEXT_PUBLIC_MAPBOX_TOKEN` is present in the environment.
- **Simulation_Engine**: The server-side singleton (`src/lib/simulation/engine.ts`) that generates algorithmic traffic observations on a configurable tick interval.
- **Tick**: One simulation cycle during which the Simulation_Engine generates new `traffic_observations` for all road segments.
- **Simulated_Clock**: The in-memory clock maintained by the Simulation_Engine, starting at 06:00 and advancing based on the speed multiplier.
- **Zone_Type**: A classification tag on a road segment (`residential`, `commercial`, `industrial`, `transit`, `highway`) that governs its traffic generation behaviour profile.
- **Scenario**: A named preset event (Rush Hour, Stadium Exodus, Major Accident, Flash Flood) that the demo operator can trigger to produce a visible, immediate effect on the map.
- **Cascade**: The propagation of increased vehicle counts to adjacent road segments when a segment reaches Gridlock.
- **Emergency_Vehicle**: An in-memory object representing a dispatched emergency vehicle travelling along a precomputed route through Meridian City intersections.
- **Signal_Preemption**: The act of forcing a traffic signal to Green for the approach direction of an Emergency_Vehicle, overriding all other signal states.
- **AI_Analyst**: The collapsible side panel that streams natural-language traffic insights generated by Gemma via Ollama.
- **Event_Feed**: The right-sidebar list of active injected events and scenario-triggered incidents.
- **Simulation_Control_Panel**: The floating bottom-left UI panel containing play/pause, speed selector, reset, and simulated time display.
- **Layer_Toggle_Panel**: The floating top-right UI panel for independently toggling map overlays (signals, incidents, predictions).
- **SSE**: Server-Sent Events — the existing real-time push mechanism used throughout STMS.
- **Meridian_City**: The fictional city whose road network is seeded into the database, centred approximately at 18.5°N, 73.8°E.
- **Road_Segment**: A named section of road in Meridian City, stored in the `road_segments` table with GeoJSON geometry.
- **Intersection**: A named junction in Meridian City, stored in the `intersections` table with latitude/longitude coordinates.
- **Dijkstra_Router**: The existing routing service (`src/lib/utils/routing.ts`) that computes shortest paths through the road network.
- **Flood_Risk_Segment**: A road segment flagged with `flood_risk = true`, eligible for Flash Flood scenario targeting.

## Requirements

### Requirement 1: Map Provider Abstraction

**User Story:** As a Traffic_Controller, I want the map to load using free OpenStreetMap tiles by default and optionally use Mapbox when a token is configured, so that the system works without any paid API keys.

#### Acceptance Criteria

1. WHEN `NEXT_PUBLIC_MAPBOX_TOKEN` is absent from the environment, THE Map_Provider SHALL initialise MapLibre GL JS with a dark OpenStreetMap tile style (Protomaps or CartoCDN dark tiles).
2. WHEN `NEXT_PUBLIC_MAPBOX_TOKEN` is present in the environment, THE Map_Provider SHALL initialise Mapbox GL JS with the `mapbox://styles/mapbox/dark-v11` style.
3. THE Map_Provider SHALL expose a unified interface with the methods `addLayer()`, `setFeatureState()`, `flyTo()`, and `on('click')` regardless of which underlying library is active.
4. THE Map_Provider SHALL be the sole module that imports MapLibre GL JS or Mapbox GL JS; no other component SHALL import either library directly.
5. WHEN the Mapbox token is absent, THE Map_Provider SHALL load MapLibre silently without displaying any error message to the user.
6. THE Map_Provider SHALL use dynamic imports with `ssr: false` because both MapLibre GL JS and Mapbox GL JS are browser-only libraries.

### Requirement 2: Road Network Data Model Extension

**User Story:** As a developer, I want the database schema to store real geographic coordinates and GeoJSON geometry for Meridian City's road network, so that the map can render accurate road positions.

#### Acceptance Criteria

1. THE `road_segments` table SHALL include a `zone_type` column storing a string array of zone classifications (`residential`, `commercial`, `industrial`, `transit`, `highway`).
2. THE `road_segments` table SHALL include a `flood_risk` boolean column defaulting to `false`.
3. THE `intersections` table SHALL already contain `latitude` and `longitude` columns (confirmed present in existing schema); THE Migration SHALL verify these are non-nullable with valid coordinate ranges.
4. THE `road_segments` table SHALL include a `geometry` column storing a GeoJSON LineString as JSON text (already present in schema; THE Migration SHALL ensure it is populated for all seeded segments).
5. THE Seed_Data SHALL contain at least 18 road segments with Meridian City names including "Central Boulevard", "North Ring Road", "Station Avenue", "Market Street", "Harbour Link", "Industrial Bypass", "Airport Expressway", "University Road", "Old Town Lane", "Tech Park Drive", "Stadium Road", and at least 7 additional named segments.
6. THE Seed_Data SHALL contain at least 11 intersections with realistic latitude/longitude coordinates centred around 18.5°N, 73.8°E.
7. THE Seed_Data SHALL assign zone_type values to all 18+ road segments.
8. THE Seed_Data SHALL flag exactly 3 road segments with `flood_risk = true`.
9. THE Seed_Data SHALL contain at least 4 traffic signals, 2 pre-existing Active incidents, congestion predictions for all segments, and route_edges forming a connected graph.
10. THE Seed_Data SHALL NOT contain any road segment named "AI Test Segment" or "Prediction Test Segment".
11. WHEN the schema migration runs, THE Migration SHALL include a corresponding Prisma migration file under `prisma/migrations/`.

### Requirement 3: Live Congestion Map Layer

**User Story:** As a Traffic_Controller, I want to see all road segments rendered on the real map with colour and width reflecting live congestion, so that I can assess the city's traffic state at a glance.

#### Acceptance Criteria

1. THE Map_Provider SHALL render all road segments as GeoJSON LineString features on the map canvas.
2. WHEN a road segment has `current_congestion = Free`, THE Map_Provider SHALL render that segment's line in colour `#22c55e`.
3. WHEN a road segment has `current_congestion = Moderate`, THE Map_Provider SHALL render that segment's line in colour `#eab308`.
4. WHEN a road segment has `current_congestion = Heavy`, THE Map_Provider SHALL render that segment's line in colour `#f97316`.
5. WHEN a road segment has `current_congestion = Gridlock`, THE Map_Provider SHALL render that segment's line in colour `#ef4444`.
6. THE Map_Provider SHALL scale line width linearly between 3px (0 vehicles) and 10px (100+ vehicles) based on the segment's current vehicle count.
7. WHEN a road segment has `current_congestion = Gridlock`, THE Map_Provider SHALL apply a visible pulse animation to that segment's line.
8. WHEN a new `traffic_observation` is written to the database, THE Map_Provider SHALL update the affected segment's colour and width within 5 seconds via the existing SSE stream.

### Requirement 4: Map Overlays

**User Story:** As a Traffic_Controller, I want to toggle independent map overlays for signals, incidents, and predictions, so that I can focus on the information most relevant to the current situation.

#### Acceptance Criteria

1. THE Layer_Toggle_Panel SHALL provide independent toggle controls for three overlays: Signal Overlay, Incident Overlay, and Prediction Overlay.
2. WHEN the Signal Overlay is enabled, THE Map_Provider SHALL render a circle marker at each intersection showing the current signal phase colour (Green = `#22c55e`, Yellow = `#eab308`, Red = `#ef4444`, Off = `#64748b`).
3. WHEN a signal phase is active, THE Map_Provider SHALL apply a pulsing dot animation to that intersection's circle marker.
4. WHEN the Incident Overlay is enabled, THE Map_Provider SHALL render a pin marker at the midpoint of each incident's road segment.
5. WHEN an incident pin marker is clicked, THE Map_Provider SHALL display a popup containing the incident type, severity, description, and time elapsed.
6. WHEN the Prediction Overlay is enabled, THE Map_Provider SHALL render road segments with a translucent dashed line style at 50% opacity showing predicted congestion for the next 60 minutes.
7. WHEN an overlay toggle is changed, THE Map_Provider SHALL show or hide the corresponding layer without reloading the page.

### Requirement 5: Map Interaction

**User Story:** As a Traffic_Controller, I want to click road segments and intersections on the map to see detailed information and trigger routing, so that I can investigate and respond to traffic conditions efficiently.

#### Acceptance Criteria

1. WHEN a road segment is clicked on the map, THE Map_Provider SHALL open a side panel displaying: segment name, current vehicle count, average speed, congestion level, active incidents, and AI prediction.
2. WHEN an intersection node is clicked on the map, THE Map_Provider SHALL display a popup containing: signal phase, last override timestamp, and AI recommended timing.
3. WHEN a road segment is right-clicked on the map, THE Map_Provider SHALL display a context menu with an "Inject Event" option.
4. WHEN the user selects two intersections in Route Mode, THE Map_Provider SHALL invoke the Dijkstra_Router and render up to 3 route options as coloured polylines on the map.

### Requirement 6: Simulation Engine

**User Story:** As a demo operator, I want a server-side simulation engine that continuously generates realistic traffic data, so that the map is always live and visually compelling during a presentation.

#### Acceptance Criteria

1. THE Simulation_Engine SHALL run as a background loop in `src/lib/simulation/engine.ts` as a server-side singleton.
2. THE Simulation_Engine SHALL generate new `traffic_observations` for all road segments on each Tick.
3. WHEN the speed multiplier is 1x, THE Simulation_Engine SHALL execute one Tick every 30 seconds.
4. WHEN the speed multiplier is 5x, THE Simulation_Engine SHALL execute one Tick every 6 seconds.
5. WHEN the speed multiplier is 10x, THE Simulation_Engine SHALL execute one Tick every 3 seconds.
6. WHEN the speed multiplier is 30x, THE Simulation_Engine SHALL execute one Tick every 1 second.
7. THE Simulation_Engine SHALL maintain one of three states: `STOPPED`, `RUNNING`, or `PAUSED`.
8. THE Simulation_Engine SHALL expose control via a `POST /api/simulation/control` endpoint accepting `{ action: "play" | "pause" | "reset", speed?: number }`.
9. THE Simulation_Engine SHALL broadcast current simulation time, speed, and state via SSE on `/api/simulation/stream`.
10. THE Simulation_Engine SHALL maintain a Simulated_Clock starting at 06:00 that advances based on the active speed multiplier.

---

### Requirement 7: Zone-Based Traffic Generation

**User Story:** As a demo operator, I want traffic generation to follow realistic zone behaviour profiles with time-of-day patterns, so that the simulation feels believable and demonstrates meaningful variation.

#### Acceptance Criteria

1. THE Simulation_Engine SHALL generate vehicle counts and speeds for `residential` zone segments that peak between 07:00–09:00 and 17:00–19:00 simulated time, and are quiet between 22:00–06:00.
2. THE Simulation_Engine SHALL generate vehicle counts and speeds for `commercial` zone segments that peak between 11:00–13:00 and 16:00–19:00 simulated time, with moderate load at other hours.
3. THE Simulation_Engine SHALL generate vehicle counts and speeds for `industrial` zone segments that maintain steady moderate load between 06:00–18:00 simulated time, and low load otherwise.
4. THE Simulation_Engine SHALL generate vehicle counts and speeds for `transit` zone segments that maintain high load during all peak hours and moderate load off-peak.
5. THE Simulation_Engine SHALL generate vehicle counts and speeds for `highway` zone segments that mirror the residential peak pattern but with higher base vehicle counts.
6. THE Simulation_Engine SHALL apply ±15% random variance to vehicle counts and speeds on each Tick.
7. THE Simulation_Engine SHALL generate vehicle counts and speeds consistent with the thresholds used by the `compute_congestion_level` stored procedure (Free: <20 vehicles or ≥80% speed limit; Moderate: 20–49 vehicles or 50–79% speed limit; Heavy: 50–79 vehicles or 20–49% speed limit; Gridlock: ≥80 vehicles or <20% speed limit).

### Requirement 8: Simulation Control Panel UI

**User Story:** As a demo operator, I want a floating control panel on the map page to play, pause, reset, and speed up the simulation, so that I can control the demo presentation in real time.

#### Acceptance Criteria

1. THE Simulation_Control_Panel SHALL be a floating panel positioned in the bottom-left corner of the map page.
2. THE Simulation_Control_Panel SHALL contain a Play/Pause toggle button, a speed selector with options 1x, 5x, 10x, and 30x, a Reset button, and a display of the current Simulated_Clock time in HH:MM format.
3. WHEN the Play/Pause toggle is activated, THE Simulation_Control_Panel SHALL call `POST /api/simulation/control` with the appropriate action.
4. WHEN the speed selector value is changed, THE Simulation_Control_Panel SHALL call `POST /api/simulation/control` with the selected speed.
5. WHEN the Reset button is clicked, THE Simulation_Control_Panel SHALL call `POST /api/simulation/control` with `{ action: "reset" }` and reset the Simulated_Clock to 06:00.
6. THE Simulation_Control_Panel SHALL be collapsible to minimise screen space usage.
7. THE Simulation_Control_Panel SHALL update the displayed Simulated_Clock time in real time via the SSE stream on `/api/simulation/stream`.

### Requirement 9: Scenario Presets

**User Story:** As a demo operator, I want to trigger named scenario presets that immediately produce dramatic, visible effects on the map, so that I can demonstrate the system's capabilities during a hackathon presentation.

#### Acceptance Criteria

1. THE Simulation_Control_Panel SHALL include a scenario dropdown with four options: Rush Hour, Stadium Exodus, Major Accident, and Flash Flood.
2. WHEN the Rush Hour scenario is triggered, THE Simulation_Engine SHALL set all `commercial` and `transit` zone segments to Heavy or Gridlock congestion for 15 simulated minutes.
3. WHEN the Stadium Exodus scenario is triggered, THE Simulation_Engine SHALL set the 5 segments nearest to "Stadium Road" to Gridlock for 20 simulated minutes, and set 3 adjacent segments to Heavy.
4. WHEN the Major Accident scenario is triggered, THE Simulation_Engine SHALL select one highway segment at random, create an Incident of type `Accident`, set that segment to Gridlock, trigger an AI anomaly alert in the AI_Analyst panel, and set 2 adjacent segments to Heavy.
5. WHEN the Flash Flood scenario is triggered, THE Simulation_Engine SHALL create `Flooding` incidents on all 3 `flood_risk` segments, set those segments to `Road_Closure` congestion equivalent (Gridlock), and re-route all active routes away from those segments.
6. THE Simulation_Engine SHALL persist scenario-triggered incidents to the database via the existing `POST /api/incidents` endpoint.
7. THE Simulation_Engine SHALL broadcast scenario state transiently via SSE; incident records SHALL be persisted to the database.

### Requirement 10: Manual Event Injection

**User Story:** As a demo operator, I want to right-click any road segment on the map and inject a traffic event, so that I can demonstrate real-time incident response during a presentation.

#### Acceptance Criteria

1. WHEN a road segment is right-clicked on the map, THE Map_Provider SHALL display a context menu containing an "Inject Event" option.
2. WHEN "Inject Event" is selected, THE Map_Provider SHALL display an event type selector with options: Accident, Road Closure, Debris, Flooding, Other.
3. WHEN an event type is confirmed, THE Map_Provider SHALL call the existing `POST /api/incidents` endpoint to create an Incident record.
4. WHEN an Incident is created via event injection, THE Simulation_Engine SHALL immediately update the affected segment's congestion to at least Heavy on the next Tick.
5. WHEN an event is successfully injected, THE Map_Provider SHALL display a visual confirmation toast notification.

---

### Requirement 11: Cascade / Ripple Logic

**User Story:** As a demo operator, I want gridlock on one segment to visibly spread to adjacent segments, so that the simulation demonstrates realistic traffic propagation.

#### Acceptance Criteria

1. WHEN a road segment reaches Gridlock, THE Simulation_Engine SHALL increase vehicle counts on all directly adjacent segments by 20% on the next Tick.
2. WHEN a Gridlock segment's adjacent segment also reaches Gridlock, THE Simulation_Engine SHALL increase vehicle counts on that segment's adjacent segments by 20% on the next Tick, up to a maximum cascade depth of 2 hops.
3. WHEN an incident on a Gridlock segment is resolved, THE Simulation_Engine SHALL return adjacent segments to their normal zone-based traffic generation on the next Tick.
4. THE Simulation_Engine SHALL implement cascade logic within the engine itself, not via database triggers.

### Requirement 12: Event Feed UI

**User Story:** As a Traffic_Controller, I want to see a live feed of active events in the map page sidebar, so that I can monitor and resolve incidents without leaving the map view.

#### Acceptance Criteria

1. THE Event_Feed SHALL be displayed in the right sidebar of the map page.
2. THE Event_Feed SHALL display each active event with: a type icon, the segment name, time elapsed since creation, a severity badge, and a Resolve button.
3. WHEN the Resolve button is clicked, THE Event_Feed SHALL call the existing incident resolution endpoint.
4. WHEN an incident has been Active for more than 2 simulated hours without resolution, THE Event_Feed SHALL display a red "ESCALATED" badge on that incident's entry.

### Requirement 13: AI Insight Panel

**User Story:** As a Traffic_Controller, I want a visible AI analyst panel that streams natural-language traffic insights, so that I can understand the current situation and communicate it to stakeholders.

#### Acceptance Criteria

1. THE AI_Analyst SHALL be a collapsible side panel titled "AI Analyst" on the map page.
2. THE AI_Analyst SHALL generate a natural-language summary every 2 minutes or on demand when the "Ask AI" button is clicked.
3. WHEN generating a summary, THE AI_Analyst SHALL include in the Ollama prompt: the top 3 congested segments, active incident count, worst predicted segment, and current Simulated_Clock time.
4. THE AI_Analyst SHALL limit prompts to a maximum of 400 tokens and responses to a maximum of 200 tokens.
5. THE AI_Analyst SHALL stream the Ollama response token-by-token via SSE to produce a visible "typing" effect in the panel.
6. THE AI_Analyst SHALL complete each response within 8 seconds of the prompt being sent.
7. IF Ollama is unavailable, THEN THE AI_Analyst SHALL display "AI Analyst offline — operating in manual mode" and continue operating without AI features.
8. THE AI_Analyst SHALL retain the last 5 insights in a scrollable history within the panel.
9. WHEN a scenario is triggered, THE AI_Analyst SHALL auto-open and generate an insight tagged with trigger type `scenario`.
10. WHEN a new Gridlock is detected on any segment, THE AI_Analyst SHALL auto-open and generate an insight tagged with trigger type `gridlock_alert`.
11. WHEN the "Ask AI" button is clicked, THE AI_Analyst SHALL generate an insight tagged with trigger type `manual`.
12. THE AI_Analyst SHALL tag each insight with one of: `scheduled`, `scenario`, `gridlock_alert`, or `manual`.

### Requirement 14: AI Signal Optimisation Visibility

**User Story:** As a Traffic_Controller, I want to see the AI's signal optimisation reasoning in the AI Analyst panel, so that I can understand and trust the automated decisions.

#### Acceptance Criteria

1. WHEN the existing Ollama signal optimisation updates a signal's timing, THE AI_Analyst SHALL display a message in the format: "Optimised [Intersection Name]: N→S green extended from Xs to Ys (confidence: Z%)".
2. THE Map_Provider SHALL include an "Optimise All Signals" button that triggers a full optimisation pass across all online signals.
3. WHEN "Optimise All Signals" is clicked, THE AI_Analyst SHALL stream all individual signal change messages into the panel as they are processed.

### Requirement 15: Emergency Dispatch Flow

**User Story:** As a Traffic_Controller, I want to dispatch an emergency vehicle along a precomputed route through Meridian City, so that I can demonstrate signal preemption and emergency priority in a visually compelling way.

#### Acceptance Criteria

1. THE Simulation_Control_Panel SHALL include an "Emergency" button with red/blue styling.
2. WHEN the Emergency button is clicked, THE Map_Provider SHALL display a modal with intersection pickers for Origin and Destination.
3. WHEN Origin and Destination are confirmed, THE Dijkstra_Router SHALL compute the fastest path ignoring congestion weights.
4. THE Simulation_Engine SHALL create an in-memory `EmergencyVehicle` object with fields: `id`, `route` (array of intersection IDs), `currentIndex` (starting at 0), and `speed` (simulated).
5. THE Emergency_Vehicle SHALL take highest priority over all manual signal overrides.

---

### Requirement 16: Signal Preemption

**User Story:** As a Traffic_Controller, I want traffic signals along the emergency route to automatically turn green as the vehicle approaches, so that the emergency vehicle can travel without stopping.

#### Acceptance Criteria

1. WHEN an Emergency_Vehicle is dispatched, THE Simulation_Engine SHALL set the signal phase to Green for the approach direction at each intersection along the route.
2. THE signal override duration for emergency preemption SHALL be 60 seconds per intersection.
3. WHEN emergency preemption is applied to a signal, THE Simulation_Engine SHALL mark the signal state as `emergency_override: true`.
4. WHEN emergency preemption is applied, THE Simulation_Engine SHALL write to the Audit_Log with action `SIGNAL_OVERRIDE_APPLY`, the intersection ID, and user `SYSTEM_EMERGENCY`.
5. WHEN the Emergency_Vehicle passes an intersection, THE Simulation_Engine SHALL revert that intersection's signal to AI-optimised timing.

---

### Requirement 17: Emergency Vehicle Map Visualisation

**User Story:** As a demo operator, I want the emergency vehicle and its route to be visually dramatic on the map, so that the feature is immediately impressive during a presentation.

#### Acceptance Criteria

1. THE Map_Provider SHALL render the emergency route as a bright blue/white animated dashed polyline.
2. THE Map_Provider SHALL render an SVG vehicle icon marker that moves along the route as the Emergency_Vehicle advances.
3. WHEN an intersection's signal is preempted, THE Map_Provider SHALL pulse that intersection's marker bright green.
4. WHEN the Emergency_Vehicle passes an intersection, THE Map_Provider SHALL return that intersection's marker to its normal signal phase colour.
5. WHILE an emergency is active, THE Map_Provider SHALL dim all non-route intersections to 50% opacity.
6. THE Map_Provider SHALL animate the emergency route polyline with a travelling pulse effect using a moving dash offset.

---

### Requirement 18: Emergency Completion

**User Story:** As a Traffic_Controller, I want the system to cleanly release all emergency overrides and notify me when the vehicle reaches its destination, so that normal operations resume automatically.

#### Acceptance Criteria

1. WHEN the Emergency_Vehicle reaches its destination intersection, THE Simulation_Engine SHALL release all emergency signal overrides.
2. WHEN the Emergency_Vehicle reaches its destination, THE Map_Provider SHALL display a "Route Cleared" overlay toast notification.
3. WHEN the Emergency_Vehicle reaches its destination, THE Simulation_Engine SHALL create an Incident record with type `Other`, description "Emergency vehicle dispatch completed", severity 1, and status `Resolved`.
4. WHEN the Emergency_Vehicle reaches its destination, THE AI_Analyst SHALL generate a narration of the completed dispatch.

### Requirement 19: Map Page as Command Centre

**User Story:** As a Traffic_Controller, I want the map page to be a full-viewport command centre with all controls as floating panels, so that the map is always the primary focus.

#### Acceptance Criteria

1. THE map page (`/map`) SHALL be the primary landing page after login for users with the `Traffic_Controller` role.
2. THE map page SHALL render the map at full viewport size with no padding and no content below the fold.
3. THE Simulation_Control_Panel SHALL be positioned in the bottom-left corner of the map page as a floating panel.
4. THE Layer_Toggle_Panel SHALL be positioned in the top-right corner of the map page as a floating panel.
5. THE right sidebar SHALL be collapsible and SHALL contain the AI_Analyst panel and the Event_Feed.
6. THE top-left corner of the map page SHALL display the STMS logo, the city name "Meridian City", and the current Simulated_Clock time.
7. THE existing sidebar navigation SHALL remain accessible but SHALL be secondary to the map view.

### Requirement 20: Monitoring Page Upgrade

**User Story:** As a Traffic_Controller, I want the monitoring page to show a data-dense sortable table by default with a global stats bar, so that I can quickly assess the entire network's state.

#### Acceptance Criteria

1. THE monitoring page SHALL display a data-dense sortable table as the default view with columns: Segment Name, Zone, Congestion (coloured badge), Vehicles, Avg Speed, Last Update, Trend (sparkline), and Actions.
2. THE monitoring page SHALL retain the existing card grid view as a toggleable alternative.
3. THE monitoring page SHALL display a global stats bar at the top showing: Total Segments, % Free, % Moderate, % Heavy, % Gridlock, and Active Incidents count.
4. WHEN a segment's data is updated via SSE, THE monitoring page SHALL briefly flash that table row to indicate the update.

### Requirement 21: Error State and Degraded Mode UI

**User Story:** As a Traffic_Controller, I want the system to degrade gracefully when components are unavailable, so that I can continue operating with whatever functionality remains.

#### Acceptance Criteria

1. WHEN the Simulation_Engine is not running, THE map page SHALL display a non-blocking "Simulation Paused" overlay badge.
2. WHEN Ollama is offline, THE AI_Analyst SHALL display its offline state message; all other features SHALL continue to function.
3. WHEN `NEXT_PUBLIC_MAPBOX_TOKEN` is absent, THE Map_Provider SHALL load MapLibre silently without displaying any error to the user.
4. WHEN an SSE connection is lost, THE client SHALL attempt to reconnect with exponential backoff for a maximum of 3 retries.
5. WHEN all 3 SSE reconnection attempts fail, THE client SHALL display a "Connection lost" toast notification.

---

### Requirement 22: Emergency Route Narration

**User Story:** As a Traffic_Controller, I want the AI Analyst to narrate the emergency vehicle's progress in real time, so that I can communicate the situation to stakeholders during a presentation.

#### Acceptance Criteria

1. WHEN an Emergency_Vehicle is dispatched, THE AI_Analyst SHALL generate a narration in the format: "Emergency vehicle on route. Clearing signals: [list of intersection names]. Estimated clearance time: X min."
2. WHEN the Emergency_Vehicle advances to the next intersection, THE AI_Analyst SHALL update the narration to reflect the current position and remaining intersections.
3. WHEN the Emergency_Vehicle reaches its destination, THE AI_Analyst SHALL generate a completion narration.
